# Feature Specification: Pre-Live Azure Infrastructure Deployment

**Feature Branch**: `001-bicep-cicd-workflow`
**Created**: 2026-03-22
**Status**: Ready for live demo baseline
**Input**: User description: "Add Bicep infrastructure-as-code CI/CD to the AI Genius project."

## Clarifications

### Session 2026-05-12

- Q: What should be ready before the live show starts? -> A: The infrastructure workflow and dev Azure resources are already deployed.
- Q: What should be created during the live show? -> A: The frontend CI/CD workflow is created from a new Spec-Kit feature and proves that code changes reach Azure Static Web Apps.
- Q: Should `deploy-infra.yml` deploy the API and frontend code? -> A: No. It provisions Azure resources only. App deployment workflows are generated by later specs.
- Q: Which authentication model should the workflow use? -> A: GitHub OIDC with `azure/login@v2`; do not use `AZURE_CREDENTIALS`.
- Q: How should the frontend workflow know what to deploy to? -> A: It uses the pre-created dev resource group and Static Web App name, then retrieves the deployment token at runtime with Azure CLI.

## User Scenarios and Testing

### User Story 1 - Prepare Azure Infrastructure Before the Live Demo (Priority: P1)

As the instructor, I need a reliable dev Azure environment before the live session starts so that the live demo can focus on Spec-Kit generating the frontend deployment workflow, not on waiting for infrastructure provisioning.

**Why this priority**: Every later deployment demo depends on the resource group, API App Service, and Static Web App already existing.

**Independent Test**: Manually run `Deploy Infrastructure to Azure` for `dev`, then verify `rg-aigenius-dev` contains an App Service Plan, API App Service, and Static Web App.

**Acceptance Scenarios**:

1. **Given** the workflow is run for `dev`, **When** it completes, **Then** `rg-aigenius-dev` exists.
2. **Given** the Bicep deployment completes, **When** the instructor opens Azure, **Then** the API App Service and Static Web App are present.
3. **Given** the workflow completes, **When** the run summary is viewed, **Then** it shows the API URL and Static Web App URL.

---

### User Story 2 - Use Keyless Azure Authentication (Priority: P1)

As the repository owner, I need the workflow to authenticate without long-lived Azure credentials so that the demo follows current cloud security practices.

**Independent Test**: Review the workflow and repository secrets. The workflow uses `azure/login@v2` with `AZURE_CLIENT_ID`, `AZURE_TENANT_ID`, and `AZURE_SUBSCRIPTION_ID`; no `AZURE_CREDENTIALS` secret is required.

**Acceptance Scenarios**:

1. **Given** the repository secrets are configured, **When** the workflow runs, **Then** Azure login succeeds through OIDC.
2. **Given** a reviewer inspects repository secrets, **When** they check for long-lived client secrets, **Then** none are required by this workflow.

---

### User Story 3 - Support Later Frontend and API Deployment Specs (Priority: P2)

As the instructor, I need the infrastructure workflow to expose stable names and URLs so later live-generated workflows can target the same Azure resources.

**Independent Test**: Check the workflow summary and Bicep outputs after a successful dev deployment.

**Acceptance Scenarios**:

1. **Given** infrastructure is deployed, **When** the frontend workflow is generated live, **Then** it can target `rg-aigenius-dev` and `aigenius-frontend-dev`.
2. **Given** the frontend is built, **When** it needs the API URL, **Then** it can use the pre-created API App Service URL.
3. **Given** the API is called from Static Web Apps, **When** the browser sends requests, **Then** CORS allows the Static Web App origin.

## Requirements

### Functional Requirements

- **FR-001**: The workflow MUST trigger on pushes to `main` and by manual `workflow_dispatch`.
- **FR-002**: The manual trigger MUST accept `environment` values `dev`, `qa`, and `prod`, defaulting to `dev`.
- **FR-003**: The workflow MUST authenticate to Azure with OIDC using `azure/login@v2`.
- **FR-004**: The workflow MUST require only `AZURE_CLIENT_ID`, `AZURE_TENANT_ID`, and `AZURE_SUBSCRIPTION_ID` repository secrets.
- **FR-005**: The workflow MUST NOT require `AZURE_CREDENTIALS`.
- **FR-006**: The workflow MUST create the target resource group if it does not exist.
- **FR-007**: The workflow MUST deploy `bicep/main.bicep` with `bicep/parameters.<environment>.json`.
- **FR-008**: The Bicep deployment MUST provision an App Service Plan, API App Service, and Static Web App.
- **FR-009**: The API App Service MUST enforce HTTPS and use the .NET runtime expected by `src/ai-genius-api`.
- **FR-010**: The API App Service MUST allow browser calls from the Static Web App origin.
- **FR-011**: The workflow MUST publish the resource group, API App Service name and URL, and Static Web App name and URL in the run summary.
- **FR-012**: All Azure resources MUST include `app`, `component`, `environment`, and `managedBy` tags.

### Key Entities

- **Infrastructure workflow**: `.github/workflows/deploy-infra.yml`, completed before the live demo.
- **Resource group**: `rg-aigenius-<environment>`, with `rg-aigenius-dev` used for the live demo.
- **API App Service**: Hosts `src/ai-genius-api` after the later API deployment workflow is generated.
- **Static Web App**: Hosts `src/ai-genius-web` after the later frontend deployment workflow is generated.
- **OIDC federated credential**: Azure trust entry for GitHub Environment subjects such as `repo:ry0y4n/ai-genius-s4-ep2-speckit:environment:dev`.

## Success Criteria

- **SC-001**: A manual `dev` infrastructure run finishes successfully before the live demo.
- **SC-002**: `rg-aigenius-dev` contains the expected App Service Plan, API App Service, and Static Web App.
- **SC-003**: The workflow run summary shows non-empty API and Static Web App URLs.
- **SC-004**: The frontend deployment spec can target the pre-created Static Web App without creating new infrastructure.
- **SC-005**: No long-lived Azure credential is required for the workflow.

## Assumptions

- The live show uses the `dev` environment only.
- QA and production parameter files remain available for demonstration and future extension but are not deployed during the live show.
- GitHub Actions is enabled for `ry0y4n/ai-genius-s4-ep2-speckit`.
- The required Azure OIDC app registration, federated credential, and repository secrets are configured before the live show.
